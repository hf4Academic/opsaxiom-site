决策树

入口：confirm_hang

[check] confirm_hang — 确认是否 hang（GPU 忙但无进展）
· 当 gpu_util > 90 → check_watchdog
· 当 gpu_util < 10 → idle_hang_exit
· 否则 → check_watchdog
⚠ NCCL hang 的典型态：某几张卡 util 100% 卡死不动(在等集合通信)，另一些 util 0(已到同步点在等别人)。整个 job 无进展但进程不退
⚠ 别急着 kill：先抓现场——py-spy dump 看卡在哪个 all_reduce、NCCL_DEBUG 日志看哪个 rank 没到、是否有 rank 因 OOM/崩溃退出导致其余 rank 死等
[check] check_watchdog — 检查是否有 NCCL 超时/某 rank 掉队
· 当 nccl_timeout > 0 → rank_stuck_exit
· 否则 → deadlock_exit
⚠ PyTorch 的 ProcessGroupNCCL watchdog timeout 说明某 rank 在超时时间内没到集合点——去找那个掉队/崩溃的 rank
[done] rank_stuck_exit — 有 NCCL/watchdog 超时，某 rank 掉队。方向：定位未到同步点的 rank(常因它先 OOM/异常退出/所在卡故障)、检查该 rank 的 GPU(转 xid-error)、确认是否单卡故障拖垮整个 job。找到掉队 ran
[done] idle_hang_exit — GPU 利用率低却无进展，疑似所有 rank 都卡在等待(集合通信死锁)或卡在非 GPU 环节(数据加载/checkpoint)。抓 py-spy dump 看主线程栈，区分是卡在 NCCL 还是卡在 IO/数据。
[done] deadlock_exit — GPU 忙但无超时记录，疑似集合通信死锁(如不同 rank 调用顺序不一致、通信组不匹配)。抓各 rank 的 py-spy dump 对比卡点、检查代码里 all_reduce/broadcast 的调用是否所有 rank 一致。
