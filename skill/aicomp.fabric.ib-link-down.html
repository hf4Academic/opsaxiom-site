决策树

入口：read_link

[check] read_link — 读取 IB 端口状态与速率
· 当 link_up == 0 → down_exit
· 当 rate_gbps < expected_rate → degraded_exit
· 当 symbol_errors > 0 and delta(symbol_errors, 60s) > 0 → error_exit
· 当 link_up == 1 and rate_gbps >= expected_rate → healthy_exit
· 否则 → escalate
⚠ IB 链路问题直接拖垮多机训练(NCCL 走 IB)。先分清是 down(完全断)、降速(协商到低速率)还是有误码(symbol errors 增长)——处置不同
⚠ State: Active 才是好的；Initializing/Down 都不通。Rate 要对上物理规格(如 HDR 200Gb/s)，降速常是线缆/光模块问题
[done] healthy_exit — IB 端口 Active 且速率达标、无误码增长。若 NCCL 仍慢，转 aicomp/collective/nccl-allreduce-slow(可能是 RoCE/拓扑/GDR 配置)而非物理链路。
[done] down_exit — IB 端口 Down/未 Active。方向：检查线缆/光模块是否插好、对端交换机端口状态、HCA 固件与驱动(OFED)、subnet manager(opensm)是否运行。链路恢复后 NCCL 自愈。
[done] degraded_exit — IB 链路 Active 但速率低于规格(降速)。方向：多为线缆/光模块质量或协商问题——换线缆/光模块、检查两端端口配置、确认物理规格一致(HDR/NDR)。降速会让集合通信带宽腰斩。
[done] error_exit — IB 链路有 symbol errors 且在增长(误码)。物理层信号质量差：清洁/更换光模块与线缆、检查弯折与距离。误码会导致重传拖慢 NCCL 甚至间歇中断。
[escalate] escalate — 打包升级：ibstat 全量、端口状态/速率/误码、opensm 状态、对端交换机端口、OFED 版本。
