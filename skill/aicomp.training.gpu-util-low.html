决策树

入口：assess

[check] assess — 确认利用率低并看波形
· 当 avg_util < 30 and util_saw_tooth > 0 → data_bound_exit
· 当 avg_util < 30 → check_bottleneck
· 当 avg_util >= 30 → acceptable_exit
· 否则 → escalate
⚠ GPU 利用率低=算力(和钱)在浪费。锯齿波形(util 忽高忽低)几乎必是数据供给跟不上——GPU 算完一批在等下一批
⚠ 别只看 utilization.gpu 的百分比：它只反映'有 kernel 在跑'，不反映算得满不满。真要看效率得看 SM 占用/MFU，但锯齿波形已足够定性数据瓶颈
[done] acceptable_exit — 利用率在可接受范围。若仍想提效，看 batch size 是否可增大、是否用了混合精度/编译优化，属调优而非故障。
[done] data_bound_exit — 利用率低且呈锯齿波形，典型数据瓶颈(GPU 等数据)。方向：增大 DataLoader num_workers 与 prefetch、 用更快的存储/本地缓存、数据预处理下放/提前、pin_memory + 异步拷贝。让数据供给跟上算力是关
[check] check_bottleneck — 非锯齿——查是通信还是小算子瓶颈
· 当 comm_time_pct > 40 → comm_bound_exit
· 当 small_kernel_pct > 40 → kernel_bound_exit
· 否则 → cpu_bound_exit
⚠ 利用率低且不锯齿：可能通信占比过高(转 nccl-allreduce-slow)、大量小算子导致 kernel launch 开销、或 CPU 前处理/同步点太多
[done] comm_bound_exit — 通信占比过高，GPU 大量时间在等集合通信。转 aicomp/collective/nccl-allreduce-slow 优化通信；或从算法侧减少通信(梯度累积、通信计算重叠)。
[done] kernel_bound_exit — 大量小 kernel，launch 开销吃掉算力。方向：算子融合(torch.compile/融合优化)、增大 batch 摊薄开销、用 CUDA Graph 减少 launch。
[done] cpu_bound_exit — 疑似 CPU 前处理/同步瓶颈。查 CPU 是否打满、是否有频繁的 .item()/.cpu() 同步点、数据增强是否在 CPU 上过重。把可并行的前处理移到 GPU 或异步化。
[escalate] escalate — 打包升级：dmon 利用率波形、通信/kernel/CPU 占比、DataLoader 配置、batch size。
