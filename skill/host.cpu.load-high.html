决策树

入口：assess_load

[check] assess_load — 判断负载是否真的过高（相对核数）
· 当 load1 > cores * 2 → classify_bottleneck
· 当 load1 <= cores * 2 → not_high
· 否则 → escalate
⚠ 负载必须除以核数看：load=16 在 4 核是灾难，在 64 核是正常。别被绝对值吓到
⚠ loadavg 计入 D 状态(不可中断睡眠)进程，高 load 不一定是 CPU 忙——可能是 IO 等待或 NFS 卡死
[done] not_high — 1 分钟负载 {{load1}} 相对 {{cores}} 核未超阈值，可能为瞬时峰值。建议观察 5/15 分钟负载趋势。
[check] classify_bottleneck — 区分 CPU 密集 / IO 等待 / 运行队列积压
· 当 count(rows[].wa > 20) >= 2 → io_bound_exit
· 当 count(rows[].us + rows[].sy > 80) >= 2 → find_top_consumers
· 当 count(rows[].r > cores * 2) >= 2 → runqueue_exit
· 否则 → find_top_consumers
⚠ wa(iowait)高→转 storage 域；us/sy 高→真 CPU 忙；r(运行队列)长但 CPU 不满→cgroup 限流或调度问题
[done] io_bound_exit — iowait 持续偏高，瓶颈在磁盘 IO 而非 CPU。请转 host/storage/iops-saturated。
[done] runqueue_exit — 运行队列长但 CPU 未打满，疑似 cgroup CPU 限流或调度问题。请查 host/cpu/throttled。
[check] find_top_consumers — 定位占用 CPU 最高的进程
· 当 rows[0].pcpu > 50 → single_hog_exit
· 当 count(rows[].pcpu > 10) >= 3 → many_procs_exit
· 否则 → escalate
⚠ pcpu 可 >100%（多核之和）。留意 fork 风暴：大量同名短命进程 ps 抓不住，用 pidstat 或 execsnoop
[done] single_hog_exit — 单进程 {{rows[0].comm}}(pid {{rows[0].pid}}) 占用最高。结合业务判断是否正常；异常则抓火焰图(perf/py-spy)定位热点函数再处置。
[done] many_procs_exit — 多进程共同拉高 CPU，疑似整体负载上升或批量任务。核对是否有定时任务/发布/流量突增。
[escalate] escalate — 打包升级：loadavg、vmstat、top 进程、核数。
