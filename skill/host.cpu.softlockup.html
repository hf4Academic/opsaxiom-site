决策树

入口：confirm_lockup

[check] confirm_lockup — 确认 lockup 类型
· 当 any(lines[] matches 'hard LOCKUP|NMI watchdog') → hard_lockup_exit
· 当 any(lines[] matches 'soft lockup') → soft_lockup_analyze
· 当 any(lines[] matches 'rcu.*stall') → rcu_stall_exit
· 当 count(lines) == 0 → no_lockup
· 否则 → escalate
⚠ soft lockup=某 CPU 在内核态跑了 >20s 没让出(watchdog 线程饿死)；hard lockup=CPU 连中断都不响应(更严重，常是硬件或死锁)
⚠ lockup 常导致系统假死/命令卡住——能登进去排查已属幸运，优先抓 dmesg 和调用栈，随时可能彻底 hang
[done] no_lockup — 近期 dmesg 无 lockup 记录。若系统曾假死，可能未留下日志(hard lockup 常来不及写盘)，检查是否有 kdump/vmcore。
[check] soft_lockup_analyze — 从调用栈判断 soft lockup 来源
· 当 any(lines[] matches 'spin_lock|mutex|down_') → lock_contention_exit
· 当 any(lines[] matches 'xfs|ext4|nfs|blk|scsi') → io_stack_exit
· 否则 → generic_soft_exit
⚠ 调用栈里的锁函数(spin_lock/mutex)→内核锁竞争或死锁；文件系统/块层函数→底层 IO 卡死(常伴坏盘/存储抖动)
[done] lock_contention_exit — soft lockup 栈指向内核锁竞争/死锁。收集完整 dmesg 栈，多为内核 bug 或驱动问题——记录内核版本，检索已知问题，考虑升级内核。禁止随意 kill 相关进程(可能加剧死锁)。
[done] io_stack_exit — soft lockup 栈指向文件系统/块层，底层 IO 卡死。检查存储健康(SMART/RAID)、多路径状态、后端存储抖动。转 host/storage 排查硬件。
[done] generic_soft_exit — soft lockup 但栈不典型。保存完整 dmesg 与内核版本，检查是否 CPU 长时间高负载饿死 watchdog(转 load-high)或已知内核 bug。
[escalate] hard_lockup_exit — hard lockup(NMI watchdog)——CPU 完全无响应，通常是硬件故障、内核死锁或虚拟化层问题。系统随时可能彻底 hang。立即：保存 dmesg、配置/检查 kdump 以便下次抓 vmcore、检查硬件(内存 ECC/
[done] rcu_stall_exit — RCU stall——某 CPU 长时间未响应 RCU 宽限期，常与 soft lockup 同源(长时间关抢占/中断)。收集栈，多指向驱动或内核 bug。
[escalate] escalate — 打包升级：dmesg lockup 记录与调用栈、内核版本、是否配置 kdump、硬件信息。
