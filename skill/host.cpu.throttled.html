决策树

入口：read_throttle

[check] read_throttle — 确认是否发生 CPU 限流及其强度
· 当 output.nr_throttled == 0 → done_not_throttled
· 当 output.throttle_ratio >= 0.25 → check_quota
· 当 output.throttle_ratio < 0.25 → done_mild
· 否则 → check_quota
⚠ cgroup v1/v2 文件路径不同：v2 是 /sys/fs/cgroup/cpu.stat，v1 是 /sys/fs/cgroup/cpu/cpu.stat，两条命令都给
⚠ throttled_usec 高但 throttle_ratio 低，是长尾尖刺型限流——看 P99 延迟而非均值，别被平均数骗过
⚠ nr_throttled==0 只说明这个 cgroup 没被限流，卡顿另有其因（转查 steal/负载/单核饱和）
[done] done_not_throttled — 未发生 CPU 限流（nr_throttled=0）。卡顿/延迟另有根因，建议转查 CPU steal、系统负载或单核饱和。
[done] done_mild — 轻度限流（限流周期占比 {{output.throttle_ratio}}，<25%）。多为突发流量下的偶发尖刺，通常无需调整；若延迟 P99 敏感，可考虑开启 CFS burst 缓冲短时突发。
[check] check_quota — 读取 CPU 配额以量化缺口
· 当 output.quota_cores > 0 → done_severe
· 当 output.quota_cores == -1 → escalate
· 否则 → escalate
⚠ quota_cores=-1 表示无限制——若无限制却在限流，说明读错了 cgroup 层级（进程实际受更上层父 cgroup 约束），升级人工核对层级
[done] done_severe — 严重限流（限流周期占比高）。当前配额约 {{output.quota_cores}} 核，明显不足。处置：K8s 下调大 Deployment 的 resources.limits.cpu（**不要直接改 cgroup 文件——kubele
[escalate] escalate — 打包升级：cpu.stat 全量（nr_periods/nr_throttled/throttled_usec）、cpu.max 配额、cgroup 版本与层级路径。
