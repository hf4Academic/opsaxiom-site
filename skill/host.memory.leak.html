决策树

入口：classify_shortage

[check] classify_shortage — 区分进程占用 / 内核 Slab / page cache 假象
· 当 slab_unreclaim_mb > mem_total * 0.2 → kernel_slab_exit
· 当 available_pct > 20 → cache_illusion_exit
· 当 available_pct <= 20 → rank_processes
· 否则 → rank_processes
⚠ 别被 used 高吓到：Linux 会拿空闲内存做 page cache，看 MemAvailable 而不是 free 列
⚠ SUnreclaim(不可回收 Slab)大 → 内核对象泄漏(如 dentry/inode)，方向完全不同于用户态进程泄漏
[done] cache_illusion_exit — MemAvailable 仍充足({{available_pct}}%)，used 高是 page cache 占用(可随时回收)，非泄漏。无需处置。
[done] kernel_slab_exit — 不可回收 Slab 占比过高，疑似内核对象泄漏。用 slabtop 看哪类对象(dentry/inode 常见于海量小文件访问)；drop_caches 只能回收可回收部分，根因需查触发内核对象暴涨的应用行为。
[check] rank_processes — 按 RSS 排序找出内存大户
· 当 rows[0].rss > mem_total * 1024 * 0.4 → single_leaker
· 当 count(rows[].rss > mem_total * 1024 * 0.1) >= 3 → distributed_exit
· 否则 → escalate
⚠ RSS 含共享内存，多进程会重复计数；判断单进程真实占用看 PSS(/proc/PID/smaps_rollup)
⚠ 确认是否泄漏要看趋势：单次快照高≠泄漏。对比历史监控或多次采样看 RSS 是否只涨不降
[done] single_leaker — 单进程 {{rows[0].comm}}(pid {{rows[0].pid}}) RSS 最高。确认其 RSS 随时间只增不减方可判定泄漏；确认后按业务窗口重启缓解，并抓 heap profile(如 jmap/pprof/valgrin
[done] distributed_exit — 多进程分散占用，无单一泄漏点，疑似整体内存规划不足或连接数/worker 数配置过高。核对进程数与单进程内存是否匹配物理内存。
[escalate] escalate — 打包升级：free、meminfo、slabtop、按 RSS 排序的进程表、内存趋势。
