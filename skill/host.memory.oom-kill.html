决策树

入口：confirm_oom

[check] confirm_oom — 确认确实发生了 OOM 及被杀进程
· 当 count(lines) >= 1 → scope_oom
· 当 count(lines) == 0 → no_oom_found
· 否则 → escalate
⚠ dmesg 时间戳可能是相对启动时间；用 dmesg -T 转成墙钟，核对是否与本次告警时间吻合
⚠ 被杀进程未必是元凶——OOM Killer 按 oom_score 挑'性价比最高'的杀，常误伤大内存的无辜进程
[done] no_oom_found — 近期 dmesg 无 OOM 记录。若应用报内存错误，可能是 cgroup 内存限制(容器 OOM)或应用自身限制，请查 /sys/fs/cgroup 或 k8s/workload/oomkilled。
[check] scope_oom — 判断是整机 OOM 还是 cgroup(容器)OOM
· 当 count(lines) >= 1 → cgroup_oom_exit
· 当 count(lines) == 0 → check_mem_trend
· 否则 → check_mem_trend
⚠ 日志里出现 memory cgroup / kubepods → 是某容器超了自己的 limit，宿主可能内存充足
[done] cgroup_oom_exit — 属于 cgroup(容器)内存超限，非宿主内存不足。请转 k8s/workload/oomkilled 或调整容器 memory limit，宿主内存正常。
[check] check_mem_trend — 判断是持续泄漏还是瞬时峰值
· 当 available_pct < 5 → still_tight_exit
· 当 available_pct >= 5 → transient_exit
· 否则 → escalate
⚠ OOM 已过去但内存现在仍紧张 → 元凶仍在运行；若已缓解 → 可能是瞬时批量任务，抓取当时的历史监控
[done] still_tight_exit — 内存当前仍紧张，元凶进程大概率仍在运行。请转 host/memory/leak 排查泄漏进程；临时缓解可考虑重启最大内存占用进程(需业务确认)。
[done] transient_exit — OOM 已缓解，内存回到正常水位，疑似瞬时峰值(批量任务/突发流量)。建议：为该主机设置内存 85% 预警；核对被杀进程是否需要调高 oom_score_adj 保护。
[escalate] escalate — 打包升级：dmesg OOM 记录、free、meminfo、被杀进程清单。
