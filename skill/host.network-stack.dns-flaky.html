决策树

入口：read_resolv

[check] read_resolv — 检查 resolv.conf 配置形态
· 当 output.nameserver_count == 0 → escalate
· 当 output.nameserver_count >= 1 → probe_servers
· 否则 → probe_servers
⚠ 多 nameserver 且 options 无 timeout 时，默认每台 5s 超时后才轮到下一台——这是'间歇性 5 秒慢'的头号元凶
⚠ 容器里 resolv.conf 是挂载进来的，改宿主机无效；K8s 默认 ndots:5 叠加多 search 域会把每次解析放大成 N 次查询
[check] probe_servers — 逐个 nameserver 探测连通与耗时
· 当 output.dead_ns_count >= 1 → done_dead_ns
· 当 output.slow_ns_count >= 1 → done_slow_ns
· 当 output.dead_ns_count == 0 and output.slow_ns_count == 0 → check_resolved
· 否则 → check_resolved
⚠ conntrack 表满时 UDP 53 丢包会伪装成 DNS 问题——若本机 conntrack 使用率高，先排查 conntrack-full
[done] done_dead_ns — 有 {{output.dead_ns_count}} 个 nameserver 不可达。由于默认串行轮询，每次解析都要先等死掉的 server 超时（默认 5s）才 fallback，表现为间歇性 5 秒慢。处置：把死的 server 从
[done] done_slow_ns — 有 {{output.slow_ns_count}} 个 nameserver 响应慢（>500ms）。多为上游 DNS 负载高或网络抖动，建议换更近的解析器或加本地缓存（dnsmasq/systemd-resolved）。
[check] check_resolved — 检查本地 DNS 缓存/systemd-resolved
· 当 service_active → done_check_resolved
· 当 not service_active → escalate
· 否则 → escalate
⚠ 所有 server 都健康却仍间歇慢，问题多在本地缓存层或应用侧连接复用——看 resolvectl statistics 的 cache miss 与 DNSSEC 失败计数
[done] done_check_resolved — 各 nameserver 均健康，systemd-resolved 在运行。间歇慢多在本地缓存/DNSSEC——检查 `resolvectl statistics` 的 cache miss 率与 DNSSEC 验证失败计数；应用侧确认是否
[escalate] escalate — 打包升级：resolv.conf 全文、各 nameserver 探测耗时、systemd-resolved 状态与 conntrack 使用率。
