决策树

入口：reproduce

[check] reproduce — 复现并区分解析层次
· 当 any(lines[] matches 'GETENT_FAIL') → check_server
· 当 not any(lines[] matches 'GETENT_FAIL') → nsswitch_exit
· 否则 → check_server
⚠ getent 走完整 nsswitch(含 /etc/hosts、mdns 等)，nslookup/dig 只问 DNS。两者结果不同能定位是 DNS 还是 hosts/缓存问题
⚠ 先确认是全解析失败还是仅某域名：某个域名解析不了 vs 所有域名都不行，根因完全不同
[done] nsswitch_exit — getent 能解析但应用报错，可能是应用自带 DNS 缓存/解析库、或 /etc/hosts 有旧记录覆盖。核对 hosts 文件与应用侧缓存。
[check] check_server — 检查配置的 DNS 服务器是否可达可用
· 当 any(lines[] matches '100% packet loss|unreachable') → server_unreachable_exit
· 当 any(lines[] matches 'SERVFAIL|REFUSED|timed out') → server_bad_exit
· 当 any(lines[] matches 'NXDOMAIN') → nxdomain_exit
· 否则 → check_resolv_sanity
⚠ systemd-resolved 环境下 resolv.conf 常指向 127.0.0.53，真正的上游要看 resolvectl status
[done] server_unreachable_exit — 配置的 DNS 服务器不可达(ping 不通)。核对网络连通性(可能是防火墙/路由问题，转 network 域)、或更换为可用 DNS。
[done] server_bad_exit — DNS 服务器返回 SERVFAIL/REFUSED/超时——服务器本身有问题或不允许本机查询。联系 DNS 管理员，或临时切换上游 DNS 验证。
[done] nxdomain_exit — DNS 明确返回 NXDOMAIN(域名不存在)。这是权威应答：核对域名拼写、该记录是否真的存在、是否内部域名但用了外部 DNS。
[check] check_resolv_sanity — 检查 resolv.conf 合理性
· 当 not any(lines[] matches 'nameserver') → no_nameserver_exit
· 否则 → escalate
⚠ resolv.conf 可能被 NetworkManager/systemd-resolved/dhclient 动态覆盖——改了不生效多半是被托管重写了
[done] no_nameserver_exit — resolv.conf 里没有有效 nameserver。核对是谁管理它(NetworkManager/resolved/dhcp)，从源头修复而非直接改文件(会被覆盖)。
[escalate] escalate — 打包升级：resolv.conf、nsswitch hosts 行、getent/nslookup 结果、上游 DNS 连通性。
