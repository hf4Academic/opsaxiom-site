决策树

入口：measure

[check] measure — 量化 TIME_WAIT 与端口占用
· 当 timewait > 28000 → locate_direction
· 当 timewait <= 28000 → not_significant
· 否则 → escalate
⚠ TIME_WAIT 是 TCP 正常机制(主动关闭方等 2MSL)，本身无害。只有当它耗尽本地端口(ip_local_port_range)导致新建外连失败时才是问题
⚠ 关键区分：是本机作为客户端主动发起短连接(出方向端口耗尽)，还是作为服务器(TIME_WAIT 在 server 侧一般不耗端口，因为四元组不同)
[done] not_significant — TIME_WAIT 数量在正常范围，未逼近端口上限。若有连接失败，方向可能在 conntrack(转 conntrack-full)或对端。
[check] locate_direction — 判断是出方向端口耗尽还是入方向堆积
· 当 rows[0].count > 20000 → outbound_exhaust_exit
· 否则 → server_side_exit
⚠ 本地端口(源端口)集中在临时端口段且逼近 range 上限 → 出方向短连接耗尽端口；这才是真问题
[done] outbound_exhaust_exit — 本机作为客户端发起海量短连接，临时端口被 TIME_WAIT 占尽，新外连报 'Cannot assign requested address'。 正解按优先级：①应用改用连接池/长连接(根治)；②扩大 ip_local
[done] server_side_exit — TIME_WAIT 主要在服务端且未耗尽端口(服务器四元组不同，一般不耗端口)，通常无需处理。若确有影响，应让客户端主动关闭连接，或用长连接减少连接创建。
[escalate] escalate — 打包升级：ss -s 汇总、TIME_WAIT 计数、本地端口段、按源端口/四元组的分布。
