决策树

入口：scope_fd

[check] scope_fd — 判断是全局 fd 耗尽还是单进程超限
· 当 allocated > max * 0.8 → rank_fd
· 当 allocated <= max * 0.8 → rank_fd
· 否则 → escalate
⚠ 两种耗尽：系统级(/proc/sys/fs/file-max)和进程级(ulimit -n / LimitNOFILE)。报 Too many open files 时先分清是哪一种
⚠ 容器里 ulimit 看的是容器的限制；systemd 服务看 LimitNOFILE 而非登录 shell 的 ulimit
[check] rank_fd — 按打开 fd 数排序找出泄漏进程
· 当 rows[0].fd_count > 10000 → classify_fd_type
· 当 rows[0].fd_count > 1000 → classify_fd_type
· 否则 → escalate
⚠ fd 数高不一定是泄漏——高并发服务器本就持有大量连接。判断泄漏看趋势(是否只增不减)和 fd 类型分布
[check] classify_fd_type — 分析泄漏 fd 的类型（socket / 文件 / pipe）
· 当 any(lines[] matches 'socket') → socket_leak_exit
· 当 any(lines[] matches 'deleted') → deleted_fd_exit
· 当 any(lines[] matches 'pipe') → pipe_leak_exit
· 否则 → file_leak_exit
⚠ socket 泄漏最常见：连接用完没 close。看 socket 是否大量 CLOSE_WAIT(对端关了本端没关)
[done] socket_leak_exit — 进程 {{pid}} 泄漏大量 socket fd。若伴随大量 CLOSE_WAIT，是应用没正确 close 连接(代码 bug)。缓解靠重启该进程，根治需研发修复连接生命周期管理。
[done] deleted_fd_exit — 大量指向已删除文件的 fd(deleted)——进程持有已 rm 的文件句柄。这也是磁盘'删了不释放空间'的元凶。缓解靠重启/reload 该进程。
[done] pipe_leak_exit — pipe/eventfd 泄漏，常见于异步框架或子进程管理 bug。交研发排查 pipe 创建后未关闭的路径。
[done] file_leak_exit — 普通文件 fd 泄漏。核对应用是否循环打开文件不关闭(如日志/配置反复 open)。重启缓解，研发修复。
[escalate] escalate — 打包升级：file-nr、ulimit、按 fd 数排序的进程、Top 进程的 fd 类型分布。
