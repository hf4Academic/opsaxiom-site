决策树

入口：count_zombies

[check] count_zombies — 统计僵尸数量并定位父进程
· 当 count(rows) > 100 → find_parent
· 当 count(rows) > 0 and count(rows) <= 100 → find_parent
· 当 count(rows) == 0 → none_now
· 否则 → find_parent
⚠ 僵尸进程本身不占 CPU/内存(只占一个 PID 表项)，真正的问题是父进程没回收——少量僵尸无害，暴涨才需处理
⚠ 危害在于耗尽 PID(nr_pids)导致 fork 失败。看 /proc/sys/kernel/pid_max 与当前 PID 用量判断紧迫性
[done] none_now — 当前无僵尸进程，可能已被回收或告警瞬时。若反复出现，观察是哪个父进程周期性产生。
[check] find_parent — 找出制造僵尸最多的父进程
· 当 rows[0].count > 50 → single_parent_exit
· 当 count(rows) > 0 → multi_parent_exit
· 否则 → escalate
⚠ ppid=1 的僵尸说明原父进程已死、被 init 接管却仍没回收——通常是 init/systemd 之外的自定义 init 有 bug
[done] single_parent_exit — 父进程 pid {{rows[0].ppid}} 制造了大量僵尸，说明它没有正确 wait() 回收子进程(常见于信号处理/多进程框架的 bug)。根治靠修复或重启该父进程——重启父进程会让 init 接管并清理僵尸。禁止 kill 僵尸本
[done] multi_parent_exit — 多个父进程各自产生少量僵尸。核对是否某类框架/脚本共性问题。整体量不大时可观察，逼近 PID 上限时优先重启制造最多的父进程。
[escalate] escalate — 打包升级：僵尸总数、pid_max 与当前用量、按父进程聚合的僵尸分布。
