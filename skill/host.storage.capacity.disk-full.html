决策树

入口：locate_mount

[check] locate_mount — 确认告警挂载点与真实使用率
· 当 rows[0].pcent >= 90 → check_inode
· 当 rows[0].pcent < 90 → false_alarm
· 否则 → escalate
⚠ 先确认监控与 df 一致：告警可能来自采集延迟，先看现值再动手
⚠ overlay/nfs 挂载的使用率语义不同，本 Skill 只处理本地文件系统
[done] false_alarm — 使用率已低于阈值
[check] check_inode — 区分空间耗尽与 inode 耗尽
· 当 rows[0].ipcent >= 95 → inode_exhaustion
· 当 rows[0].ipcent < 95 → check_deleted_open
· 否则 → escalate
⚠ df 显示有空间但应用报 No space left 时，几乎总是 inode 耗尽——这是本步骤存在的原因
[check] check_deleted_open — 检查已删除但仍被进程打开的文件
· 当 output.value > mount.size * 0.2 → ask_restart_holder
· 当 output.value <= mount.size * 0.2 → find_big_files
· 否则 → find_big_files
⚠ 典型元凶：日志被 rm 但进程未 reopen。rm 不释放空间，重启/reload 持有进程才释放
[ask] ask_restart_holder — 确认是否可以重启/重载占用进程
[action] reload_holder — 重载持有已删除文件的进程
[check] find_big_files — 定位可疑大文件与增长目录
· 当 count(rows[].path matches '/var/log') >= 1 → compress_old_logs
· 当 count(rows) > 0 → ask_quarantine
· 否则 → escalate
⚠ 只看不删。任何删除决定都必须经过人（ask 节点），Agent 无权判断业务文件价值
[action] compress_old_logs — 压缩 7 天前的历史日志（完全可逆）
[ask] ask_quarantine — 人工确认可移出的文件
[action] quarantine_files — 移动选定文件到隔离区（删除的可逆改造）
⚠ 隔离区在同一文件系统时 mv 是 O(1)；隔离区规划必须与业务盘同 FS，否则大文件移动本身会加剧写入
⚠ 7 天后 Agent 会提示确认清理——最终删除永远是人的动作（R1）
[check] inode_exhaustion — 定位海量小文件目录
· 当 count(rows) > 0 → ask_quarantine
· 否则 → escalate
⚠ 典型元凶：sessions 目录、mail 队列、未清理的临时文件。处置同样走隔离区，不直接删
[check] final_check — 复核使用率并给出防复发建议
· 当 rows[0].pcent < 85 → done_ok
· 当 rows[0].pcent >= 85 → escalate
· 否则 → escalate
[done] done_ok — 使用率已降至 {{rows[0].pcent}}%。防复发建议：为元凶目录配置 logrotate/定期清理； 为该挂载点设置 85% 预警（早于 90% 告警）；若隔离区有文件，7 天后需人工确认清理。
[escalate] escalate — 已采集的事实与走过的路径将打包升级。常见的树外原因：配额（quota）、reserved blocks、文件系统错误。
