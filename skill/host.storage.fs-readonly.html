决策树

入口：confirm_ro

[check] confirm_ro — 确认挂载点确为只读及触发原因
· 当 any(lines[] matches '(^|,)ro(,|$)') → find_cause
· 当 not any(lines[] matches '(^|,)ro(,|$)') → not_ro
· 否则 → escalate
⚠ 文件系统变只读几乎总是内核的自我保护：检测到 IO 错误或元数据损坏后强制 RO，避免继续写坏数据
⚠ 此时最忌讳的操作就是无脑 remount rw ——如果底层是坏盘或元数据损坏，会把数据彻底写烂
[done] not_ro — 当前挂载已是可读写，可能问题已被处理或告警延迟。核对 dmesg 是否有历史 RO 记录。
[check] find_cause — 判断根因是硬件 IO 错误还是文件系统逻辑错误
· 当 any(lines[] matches 'I/O error|medium error') → hardware_exit
· 当 any(lines[] matches 'EXT4-fs error|corruption|metadata') → assess_disk
· 否则 → escalate
⚠ I/O error / medium error → 磁盘硬件故障，别 fsck(会加速坏盘死亡)，先备份可读数据再换盘
⚠ EXT4-fs error / corruption → 文件系统逻辑损坏，需离线 fsck 修复
[escalate] hardware_exit — 根因是底层磁盘硬件 IO 错误。禁止 remount/fsck。立即：抢救性备份该盘仍可读的数据(dd/ddrescue)、检查 RAID 是否降级、准备换盘。已生成硬件故障处置清单。
[check] assess_disk — 判断该文件系统能否离线修复
· 当 any(lines[] matches 'IS_ROOT') → root_fs_exit
· 当 any(lines[] matches 'NOT_ROOT') → offline_fsck
· 否则 → escalate
[escalate] root_fs_exit — 根文件系统损坏无法在线卸载修复。需要在维护窗口通过 救援模式/单用户模式 或从 initramfs 跑 fsck。请安排停机窗口，已生成 rescue 修复步骤清单。
[action] offline_fsck — 卸载并对数据盘执行 fsck 修复
⚠ human_only：Agent 永不代执行本步骤，只输出指导。fsck 是破坏性修复，人必须在场并已确认备份
⚠ fsck 前若无块级快照/备份，不要开始——fsck -y 会自动删改损坏结构，无法撤销单条修改
[done] done_ok — fsck 修复完成且已重新挂载为 rw，无新错误。建议：核对 lost+found 是否有被移出的文件；对该盘做一次 SMART 健康检查排除硬件隐患。
[escalate] escalate — 打包升级：mount 选项、dmesg fs/IO 错误、设备型号、RAID/SMART 状态、是否为根分区。
