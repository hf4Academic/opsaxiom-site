决策树

入口：confirm_saturation

[check] confirm_saturation — 确认磁盘是否真的饱和
· 当 count(rows[].util > 90) >= 2 → latency_vs_throughput
· 当 count(rows[].util > 90) < 2 → not_saturated
· 否则 → escalate
⚠ %util 对 SSD/NVMe 会误导：SSD 内部并行，util=100% 不代表打满，要结合 await 和实际 IOPS 判断
⚠ 先确认是哪块盘：多盘系统别把系统盘和数据盘搞混
[done] not_saturated — 磁盘利用率未持续超 90%，非 IO 饱和。若应用仍慢，检查是否单次大 IO 或 fsync 频繁(await 高但 util 不高)。
[check] latency_vs_throughput — 区分高延迟 vs 高吞吐 vs 高 IOPS
· 当 count(rows[].await > 50) >= 2 → slow_disk_exit
· 当 count(rows[].aqu_sz > 10) >= 2 → find_io_process
· 否则 → find_io_process
⚠ await 高=每次 IO 慢(盘老化/RAID 降级/介质问题)→查硬件；aqu-sz 高=请求太多→找发起方
[done] slow_disk_exit — 每次 IO 延迟高(await 大)但队列不深，指向磁盘介质本身慢：盘老化、RAID 降级重建中、SSD 掉速或坏道。查 SMART/RAID 状态与 dmesg IO 错误，转 host/storage/raid-degraded 或换盘
[check] find_io_process — 定位产生 IO 的进程
· 当 rows[0].io_pct > 50 → single_io_hog
· 当 count(rows) > 0 → many_io_exit
· 否则 → escalate
⚠ iotop 需 root 且需内核开 CONFIG_TASK_IO_ACCOUNTING；没有则退回 pidstat -d 或 /proc/PID/io
⚠ 刷脏页由 kworker/flush 线程代做，元凶可能显示为内核线程——要往上追是谁在写
[done] single_io_hog — 单进程 {{rows[0].comm}} 主导 IO。结合业务判断：备份/日志/大查询是否可错峰或限速(ionice/cgroup blkio)。持续异常则抓 IO 栈(biosnoop/blktrace)。
[done] many_io_exit — 多进程分散 IO，疑似整体负载或存储性能不足。核对是否有批量任务叠加；评估存储是否需扩容/换 SSD。
[escalate] escalate — 打包升级：iostat -dx、按 IO 排序的进程、设备型号、RAID 状态。
