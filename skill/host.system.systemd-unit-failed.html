决策树

入口：read_state

[check] read_state — 读取单元状态与失败原因
· 当 result == 'exit-code' → read_logs
· 当 result == 'timeout' → timeout_exit
· 当 result == 'oom-kill' → oom_exit
· 当 active_state == 'active' → already_ok
· 否则 → read_logs
⚠ 先看 Result 再动手：exit-code 是程序自己退了，timeout 是启动超时，oom-kill 要转内存排查
⚠ 别急着 restart：如果是配置错误，重启只会再失败一次，先看日志定位根因
[done] already_ok — 服务当前 active，可能已恢复或告警延迟。核对 systemctl 是否有历史失败计数(NRestarts)。
[done] timeout_exit — 启动超时。检查 TimeoutStartSec 是否过短、依赖服务(After/Requires)是否就绪、是否卡在网络/挂载等待。日志里看卡在哪一步。
[done] oom_exit — 服务被 OOM 杀死。请转 host/memory/oom-kill，或调整该单元的 MemoryMax/MemoryHigh。
[check] read_logs — 从 journal 读取失败细节
· 当 any(lines[] matches 'Permission denied|EACCES') → perm_exit
· 当 any(lines[] matches 'No such file|not found|cannot open') → missing_exit
· 当 any(lines[] matches 'Address already in use|bind') → port_exit
· 当 any(lines[] matches 'config|parse|syntax|invalid') → config_exit
· 否则 → try_restart
⚠ journal 里的第一条 error 通常是根因，后面的往往是连锁反应，从前往后读
[done] perm_exit — 权限问题(EACCES)。核对运行用户(User=)、文件/目录属主、SELinux/AppArmor 上下文。
[done] missing_exit — 缺文件/路径。核对 ExecStart 路径、WorkingDirectory、依赖的配置或数据文件是否存在。
[done] port_exit — 端口被占用(Address already in use)。用 ss -ltnp 找占用进程；确认是否上一个实例未退干净或端口冲突。
[done] config_exit — 配置解析错误。核对最近的配置变更；多数服务有 -t/--test 校验子命令，先离线验证配置再启动。
[action] try_restart — 排除配置类错误后尝试重启一次
⚠ 仅在日志未指向明确配置/权限错误时才走重启——盲目重启掩盖真因还可能触发 restart 风暴
[done] done_ok — 服务已恢复 active。若 NRestarts 计数偏高，排查是否存在周期性 crash 需要研发介入。
[escalate] escalate — 打包升级：systemctl show、journalctl 错误、单元文件、依赖树。
