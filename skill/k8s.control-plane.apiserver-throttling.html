决策树

入口：check_flowcontrol

[check] check_flowcontrol — 查 apiserver 优先级与公平队列
· 当 output.value > 0 → check_who
· 当 output.value == 0 → done_no_reject
· 否则 → check_who
⚠ 1.20+ 用 APF(API Priority and Fairness)做限流，rejected 计数说明有请求被丢；别再靠老的 --max-requests-inflight 判断
[check] check_who — 看是否某控制器/客户端狂发请求
· 当 output.value > 1000 → done_client_storm
· 当 output.value <= 1000 → done_apf_tune
· 否则 → escalate
⚠ 失控的 controller/operator(reconcile 死循环)或大量 list-watch 是 apiserver 压力主因；先找狂发请求的源，而非一味调大 apiserver 配额
[done] done_no_reject — APF 无拒绝。kubectl 慢另查(网络/etcd/大对象 list)。
[done] done_client_storm — 疑似某客户端/控制器请求风暴。用 apiserver 审计日志或 APF 的 flow schema 定位高频来源，修复其请求行为(加缓存/降频/修死循环)。
[done] done_apf_tune — 请求量不算极端但仍被限。可为关键工作负载调整 APF 的 FlowSchema/PriorityLevelConfiguration 提高其配额，保障核心请求。
[escalate] escalate — 打包升级：已采集的事实与走过的路径。
