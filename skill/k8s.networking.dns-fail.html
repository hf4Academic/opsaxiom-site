决策树

入口：scope_dns

[check] scope_dns — 判断是全集群 DNS 挂还是单 Pod 问题
· 当 count(lines) == 0 → coredns_gone_exit
· 当 any(lines[] matches 'false|CrashLoop|Pending') → coredns_unhealthy_exit
· 否则 → test_resolution
⚠ 先看 CoreDNS 本身健康不健康：它挂了→全集群解析瘫痪(大面积故障)；它好好的→是单 Pod/单域名问题(小范围)
⚠ CoreDNS 通常多副本跑在不同节点，个别副本挂不致命；全挂或都不 Ready 才是灾难
[escalate] coredns_gone_exit — 找不到 CoreDNS Pod——DNS 组件缺失(被误删/部署异常)。这是集群级严重故障，全集群服务发现瘫痪。立即检查 kube-dns Deployment 是否存在、是否被驱逐。已生成 CoreDNS 恢复清单。
[done] coredns_unhealthy_exit — CoreDNS Pod 不健康(CrashLoop/Pending/NotReady)，导致集群解析故障。查 CoreDNS 日志(常见：Corefile 配置错误、上游 DNS 环路、内存不足被 OOM)。转对应 workload Ski
[check] test_resolution — CoreDNS 健康——从 Pod 内测试解析
· 当 any(lines[] matches 'connection timed out|no servers') → pod_dns_config_exit
· 当 any(lines[] matches 'NXDOMAIN|can.t resolve') → search_domain_exit
· 当 any(lines[] matches 'Address|answer') → partial_ok_exit
· 否则 → escalate
⚠ 从出问题的 Pod 里测，而不是随便找个 Pod：DNS 问题常与特定节点/网络路径相关。测 kubernetes.default 是最基础的连通性验证
⚠ exec 假设容器内有 nslookup——distroless/精简镜像没有。失败时改用 `kubectl debug -it {{pod}} --image=nicolaka/netshoot -- nslookup ...` 注入临时排查容器，或用同节点 DaemonSet 探针
[done] pod_dns_config_exit — Pod 连不上 DNS 服务(超时)。核对 Pod 的 /etc/resolv.conf nameserver 是否指向 kube-dns ClusterIP、该 ClusterIP 是否可达(kube-proxy/iptables 规则)
[done] search_domain_exit — 能连 DNS 但解析返回 NXDOMAIN。核对：短名依赖的 search 域(ndots)是否正确、跨 namespace 是否用了全限定名(svc.ns.svc.cluster.local)、外部域名是否 CoreDNS 转发上游失败。
[done] partial_ok_exit — 基础解析正常，问题可能限于特定域名或间歇性。检查目标 Service 是否存在、是否 CoreDNS 缓存/上游间歇故障、是否个别 CoreDNS 副本异常导致部分请求失败。
[escalate] escalate — 打包升级：CoreDNS Pod 状态与日志、从故障 Pod 的 nslookup 结果、resolv.conf、kube-dns Service。
