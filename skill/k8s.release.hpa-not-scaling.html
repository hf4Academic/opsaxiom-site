[check] read_metrics — 检查 HPA 是否拿得到指标
· 当 any(lines[] matches 'unknown|<unknown>|FailedGetMetrics|unable to') → metrics_unavailable_exit
· 当 any(lines[] matches 'ScalingLimited|TooManyReplicas') → max_reached_exit

⚠ HPA 不扩容第一步永远是看它有没有指标：显示 <unknown> 就是 metrics-server 或自定义指标源没数据，此时 HPA 完全瘫痪(不敢扩不敢缩)
⚠ HPA 有稳定窗口(默认扩容立即、缩容 5 分钟)和容忍度(10%)——指标刚过阈值一点点、或刚扩过，不动是正常的

[done] max_reached_exit — HPA 已到 maxReplicas 上限(ScalingLimited)。这是配置约束不是故障——若确需更多副本，调高 maxReplicas；同时核对是否底层资源(节点)也够，否则扩了也是 Pending。
[check] check_threshold — 有指标——判断是否真该扩容
· 当 current_util < target_util → below_threshold_exit
· 当 current_replicas >= max_replicas → max_reached_exit
· 当 current_util >= target_util and current_replicas < max_replicas → should_scale_exit
· 否则 → escalate
⚠ 当前利用率没超目标阈值→本来就不该扩，这不是 bug。确认你看的指标口径与 HPA 一致(HPA 按 requests 算利用率，不是按 limit 或绝对值)
[done] below_threshold_exit — 当前指标未达扩容阈值，HPA 不扩是正确行为。若你认为该扩，检查：阈值是否设得过高、HPA 依据的指标是否真反映负载(如按 CPU 但瓶颈其实在内存/连接数——应加对应指标)。
[done] should_scale_exit — 指标已超阈值且未达上限却没扩，异常。检查 HPA behavior 的 stabilizationWindow/policies 是否过于保守、是否刚扩过在冷却、controller-manager 是否正常。查 HPA events 看是
[escalate] escalate — 打包升级：HPA describe(metrics/conditions/events)、metrics-server 状态、Pod requests 配置、目标与当前指标。
