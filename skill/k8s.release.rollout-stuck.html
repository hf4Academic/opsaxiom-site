决策树

入口：locate_stuck

[check] locate_stuck — 定位卡在哪个环节
· 当 updated_replicas == 0 → new_pods_not_starting
· 当 updated_replicas > 0 and ready_replicas < spec_replicas → new_pods_not_ready
· 否则 → check_pdb
⚠ 先看新 RS 的 Pod 有没有起来：updatedReplicas=0→新 Pod 根本没创建(配额/调度问题)；>0 但没 Ready→新 Pod 起来了但过不了就绪探针
⚠ 发布卡住时旧版本通常还在服务(这是 rollingUpdate 的保护)，所以业务可能没受影响——别慌，有时间慢慢排
[check] new_pods_not_starting — 新 Pod 未创建——查为什么
· 当 any(lines[] matches 'quota|exceeded|forbidden') → quota_exit
· 当 any(lines[] matches 'FailedCreate|limit') → quota_exit
· 否则 → scheduling_exit
⚠ 新 RS 建了但 Pod 数为 0，看 RS 的 events：常见是 ResourceQuota 超限或 LimitRange 拒绝
[done] quota_exit — 新 Pod 因配额/限额创建失败(ResourceQuota 或 LimitRange)。核对 namespace 配额余量与新版本 requests；发布若调高了资源请求会触发。方向：清理资源或调整配额/请求。
[done] scheduling_exit — 新 Pod 创建了但调度不上去。转 k8s/scheduling/pending-unschedulable 排查调度失败原因(资源/污点/亲和性)。
[check] new_pods_not_ready — 新 Pod 起来了但不 Ready——查就绪失败
· 当 any(lines[] matches 'CrashLoop|Error') → crashloop_exit
· 当 any(lines[] matches 'Running.*false') → readiness_exit
· 否则 → escalate
⚠ 新版本 Pod 起不来时，progressDeadlineSeconds 到期后 rollout 会标记为失败但不自动回滚——需要人工决定 fix-forward 还是回滚
[done] crashloop_exit — 新版本 Pod CrashLoop，发布引入了崩溃。转 k8s/workload/crashloop 定位崩溃原因。决策：能快速修则 fix-forward，否则 kubectl rollout undo 回滚(转 k8s/release/
[done] readiness_exit — 新 Pod Running 但 readinessProbe 不过，卡在就绪。检查新版本的健康检查端点、启动耗时是否超过探针初始延迟、依赖是否就绪。转 k8s/workload/probe-failing。
[check] check_pdb — 检查是否被 PodDisruptionBudget 阻塞
· 当 any(lines[] matches '0 +[0-9]|DISRUPTIONS') → pdb_exit
· 否则 → escalate
⚠ PDB 设置过严(minAvailable 太高)会阻止旧 Pod 被驱逐，导致滚动更新推不下去——容易被忽略的卡点
[done] pdb_exit — PodDisruptionBudget 阻止了旧 Pod 驱逐，滚动更新卡住。核对 PDB 的 minAvailable/maxUnavailable 是否与副本数矛盾(如 3 副本却要求 minAvailable=3)。
[escalate] escalate — 打包升级：deploy status、新老 RS 状态、Pod 就绪情况、events、PDB。
