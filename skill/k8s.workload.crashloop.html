决策树

入口：read_exit

[check] read_exit — 读取上次退出原因与退出码
· 当 exit_code == 137 → oom_or_killed
· 当 exit_code == 143 → sigterm_exit
· 当 exit_code == 1 or exit_code == 2 → read_app_logs
· 当 exit_code == 0 → premature_exit
· 否则 → read_app_logs
⚠ 退出码是第一线索：137=被 SIGKILL(常为 OOM)，143=SIGTERM，非0=应用自身错误，0=进程正常退出但 K8s 期望它常驻(命令写错)
⚠ 看的是 lastState.terminated（上一次挂的原因），不是 currentState——CrashLoop 时容器正在重启间隙
[check] oom_or_killed — 退出码 137——区分 OOM 还是被主动杀
· 当 any(lines[] matches 'OOMKilled') → oom_exit
· 否则 → liveness_kill_exit
[done] oom_exit — 容器 OOMKilled(超过 memory limit)。方向：调高 resources.limits.memory，或排查应用内存泄漏。转 k8s/workload/oomkilled 深入。
[done] liveness_kill_exit — 137 但非 OOM，多为 liveness probe 失败导致 kubelet 杀容器重启。检查 livenessProbe 配置是否过严(初始延迟太短/超时太紧)、应用启动是否慢于探针容忍。
[done] sigterm_exit — 收到 SIGTERM 后退出(143)。检查是否被驱逐/滚动更新，或应用未正确处理优雅关闭。若非预期，查 events 看谁发的信号。
[done] premature_exit — 进程退出码 0 但 Pod 期望常驻——通常是启动命令/entrypoint 执行完就结束了(如误把一次性脚本当常驻服务)。核对 command/args 与镜像 CMD。
[check] read_app_logs — 读取应用日志定位崩溃原因
· 当 any(lines[] matches 'panic|fatal|exception|traceback') → app_error_exit
· 当 any(lines[] matches 'connection refused|timeout|no such host') → dependency_exit
· 当 any(lines[] matches 'permission denied|config|not found') → config_exit
· 否则 → escalate
⚠ 关键是 --previous：崩溃的是上一个容器实例，不加 --previous 看到的是新实例(可能还没崩或没日志)
[done] app_error_exit — 应用代码级错误(panic/exception)。日志已定位崩溃栈，交研发修复。运维侧确认非环境问题(配置/依赖)即可。
[done] dependency_exit — 启动时连不上依赖(数据库/下游服务)导致崩溃。核对依赖服务是否就绪、Service/DNS 是否可解析、NetworkPolicy 是否放行。可能需要 initContainer 等待依赖或应用加重试。
[done] config_exit — 配置/权限问题导致启动失败。核对 ConfigMap/Secret 是否挂载正确、文件路径、运行用户权限(securityContext)、只读文件系统限制。
[escalate] escalate — 打包升级：退出码与 reason、--previous 应用日志、events、resources 配置、探针配置。
