决策树

入口：check_notready

[check] check_notready — 查有多少 Pod 卡在未 Ready
· 当 output.value > 0 → check_events
· 当 output.value == 0 → done_all_ready
· 否则 → check_events
⚠ Running 但 Ready 0/1=容器起来了但 readiness 探针没过——Service 只把流量给 Ready 的 Pod,所以'进程在却没流量'多半是 readiness 卡住,不是进程挂了
[check] check_events — 看是否有 readiness 探针失败事件
· 当 output.value > 0 → done_probe_fail
· 当 output.value == 0 → done_ready_gate
· 否则 → escalate
⚠ readiness 探针失败常因:探针路径/端口配错、应用启动慢但 initialDelay 太短、依赖(DB/下游)没就绪应用自报未 ready——看应用日志确认它为啥说自己没好
[done] done_all_ready — Pod 均 Ready。流量不进另查(Service selector/端口/kube-proxy/NetworkPolicy)。
[done] done_probe_fail — readiness 探针在失败。核对探针配置(路径/端口/协议)、initialDelaySeconds 是否够应用启动、应用是否因依赖未就绪自报未 ready;修正后 Pod 进入 Ready 才会被加入 endpoints。
[done] done_ready_gate — 未 Ready 但无探针失败事件,可能是 readinessGates/自定义条件未满足或探针刚开始。describe Pod 看 Conditions 与 readinessGates,确认卡在哪个门槛。
[escalate] escalate — 打包升级：已采集的事实与走过的路径。
