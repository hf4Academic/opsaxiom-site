决策树

入口：measure

[check] measure — 确认 URP 规模
· 当 urp_count > 0 → check_brokers
· 当 urp_count == 0 → healthy_exit
· 否则 → escalate
⚠ URP=ISR(同步副本集)小于副本数，意味着容错余量下降——再挂一个副本可能丢数据或分区不可用。这是与时间赛跑
⚠ 先看 URP 是集中在某个 broker(该 broker 出问题)还是分散(集群级问题如网络/负载)
[done] healthy_exit — 无 under-replicated 分区，副本同步健康。若曾告警，可能是短暂抖动已恢复。
[check] check_brokers — 判断是单 broker 问题还是集群级
· 当 brokers_down > 0 → broker_down_exit
· 否则 → replication_slow_exit
⚠ 有 broker 下线 → URP 是必然结果，先恢复 broker；broker 都在但仍 URP → 副本同步慢(网络/磁盘/负载)
[done] broker_down_exit — 有 broker 下线导致其上副本脱离 ISR。方向：恢复下线 broker(查其进程/磁盘/GC)。恢复后副本会自动追平重新入 ISR。副本恢复期间避免再重启其他 broker。
[done] replication_slow_exit — broker 都在但副本同步慢(follower 追不上 leader)。方向：查 follower broker 的磁盘 IO/网络/GC 是否瓶颈、replica.lag.time.max.ms 是否过小误踢、是否某分区流量过大。转 h
[escalate] escalate — 打包升级：URP 分区清单、各 broker 状态、ISR 变化、follower 资源(IO/网络/GC)、分区流量分布。
