决策树

入口：measure

[check] measure — 量化连接使用率
· 当 connected > max_connections * 0.9 → analyze
· 当 connected <= max_connections * 0.9 → not_full
· 否则 → escalate
⚠ 报 Too many connections 时，先确认是真的高并发还是连接泄漏(Sleep 态连接堆积)——两者解法相反
⚠ 别急着调大 max_connections：如果是泄漏，调大只是推迟爆炸，还会加重内存(每连接有开销)
[done] not_full — 连接使用率未超 90%。若曾报错，可能瞬时突发已过；核对连接池配置是否合理。
[check] analyze — 区分泄漏(Sleep 堆积) vs 真实高并发
· 当 long_sleep > max_connections * 0.3 → leak_exit
· 否则 → high_concurrency_exit
⚠ 大量 Sleep 且 TIME 很大 → 应用连接用完不释放(连接池 bug/未关连接)，这是泄漏
[done] leak_exit — 大量长时间 Sleep 连接，疑似应用连接泄漏(连接池配置错误或代码未释放)。方向：查应用连接池 max/idle 配置、排查未 close 的连接。临时缓解可调低 wait_timeout 让 MySQL 主动回收空闲连接。根治在应用侧。
[done] high_concurrency_exit — 连接以活跃查询为主，是真实高并发。方向：引入/优化连接池复用连接、评估读写分离分流、必要时适度调大 max_connections(评估内存)。
[escalate] escalate — 打包升级：Threads_connected、max_connections、按 COMMAND/TIME 聚合的连接分布、应用连接池配置。
