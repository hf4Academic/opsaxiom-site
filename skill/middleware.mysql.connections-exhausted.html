[check] check_conn — 确认连接使用率
· 当 output.conn_ratio >= 0.9 → who_holds
· 当 output.conn_ratio < 0.9 → done_not_exhausted
· 否则 → who_holds
⚠ 当前未满不代表没问题：若报错发生在历史时段，看 Connection_errors_max_connections 累计计数是否增长
⚠ 上调 max_connections 前先看 `ulimit -n`：文件句柄不够时 MySQL 会静默 clamp 实际连接上限
[done] done_not_exhausted — 当前连接使用率 {{output.conn_ratio}}，未打满。若确有 Too many connections 报错，多为历史峰值瞬时——建议观察 Connection_errors_max_connections 计数并配置连接数预
[check] who_holds — 定位连接被谁占用
· 当 output.sleep_ratio >= 0.5 → done_kill_idle
· 当 output.sleep_ratio < 0.5 and output.conn_total > 0 → act_raise_max
· 否则 → escalate
⚠ 5.7 与 8.0 的 processlist 表位置不同：8.0 用 performance_schema.processlist（省全局锁），5.7 用 information_schema.processlist
[done] done_kill_idle — 空闲(Sleep)连接占大头（{{output.sleep_ratio}}），典型的应用连接池泄漏——连接借出不还。 **治本是修应用连接池配置（maxIdle/idleTimeout）**。临时缓解：生成 KILL 清单人工执行—— `S
[action] act_raise_max — 临时上调 max_connections（活跃连接确实高）
⚠ SET GLOBAL 重启后失效——永久生效需写 my.cnf 的 [mysqld] max_connections 并提示，否则重启打回原形
[done] done_raised — 已临时上调 max_connections 至 512（回滚基准 {{max_connections_before}}）。这是止血不是治本——同时排查为何活跃连接这么高（慢查询堆积/连接池过大/突发流量），并把新值写入 my.cnf 永久化
[escalate] escalate — 打包升级：Threads_connected/max_connections、按 user·host·command 聚合的连接分布、Connection_errors_max_connections、内存水位与 ulimit -n。
