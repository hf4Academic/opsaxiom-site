决策树

入口：check_waits

[check] check_waits — 当前是否存在锁等待
· 当 lock_waits > 0 → find_blocker
· 当 lock_waits == 0 → history_exit
· 否则 → escalate
⚠ 死锁(deadlock)和锁等待超时(lock wait timeout)是两回事：死锁 InnoDB 会自动回滚一方并记入 SHOW ENGINE INNODB STATUS；锁等待超时是等太久报错。先分清
⚠ 当前无等待≠没问题：死锁往往瞬时发生后自动解开，要看 SHOW ENGINE INNODB STATUS 的 LATEST DETECTED DEADLOCK 复盘
[done] history_exit — 当前无锁等待。若业务报死锁，看 SHOW ENGINE INNODB STATUS 的 LATEST DETECTED DEADLOCK 段，分析两个事务的加锁顺序，通常是加锁顺序不一致导致。交研发统一加锁顺序。
[check] find_blocker — 定位阻塞源事务
· 当 long_blocker > 0 → long_txn_exit
· 否则 → contention_exit
⚠ 阻塞源若是长事务(trx_started 很早)，多为忘了提交/应用 hang 住持锁不放——这是最典型的锁等待元凶
⚠ 8.0 用 performance_schema.data_lock_waits；5.7 无此表，改用 information_schema.INNODB_LOCK_WAITS + INNODB_TRX 关联
[done] long_txn_exit — 存在长事务持锁不放，阻塞其他事务。查该事务 SQL 与来源(是否应用未提交/事务过大)。方向：修复应用事务管理、缩小事务、必要时人工 kill 阻塞源事务(需确认其未提交的写入影响)。
[done] contention_exit — 锁等待但无明显长事务，属高并发下的热点行竞争。方向：优化事务缩短持锁时间、减少热点行更新(如计数器分桶)、调整加锁顺序避免死锁。
[escalate] escalate — 打包升级：data_lock_waits、INNODB_TRX 长事务、SHOW ENGINE INNODB STATUS 死锁段、涉及的表与索引。
