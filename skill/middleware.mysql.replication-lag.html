决策树

入口：read_lag

[check] read_lag — 读取延迟与复制线程状态
· 当 io_running == 'No' or sql_running == 'No' → broken_exit
· 当 seconds_behind > 60 → analyze_lag
· 当 seconds_behind <= 60 → ok_exit
· 否则 → escalate
⚠ Seconds_Behind_Master 只在 SQL 线程运行时有意义；线程停了它显示 NULL——先确认两个线程都 Yes 再谈延迟
⚠ 延迟大先分清是 IO 线程拉不过来还是 SQL 线程回放不过来，两者根因不同
[done] broken_exit — 复制线程已停(IO 或 SQL 为 No)，不是延迟是中断。转 middleware/mysql/replication-broken 查 Last_Error。
[done] ok_exit — 延迟 {{output.seconds_behind}}s 在可接受范围。若业务对一致性敏感，考虑读写分离时的延迟容忍或半同步。
[check] analyze_lag — 判断延迟来源（回放慢 / 大事务 / 单线程瓶颈）
· 当 parallel_workers <= 1 → single_thread_exit
· 否则 → big_txn_exit
⚠ 从库回放默认单线程，主库并发写入时从库追不上是常态。开并行复制(slave_parallel_workers)是主要解法
⚠ replication_applier_status_by_worker 是 8.0 表；5.7 并行复制看 SHOW SLAVE STATUS 与 performance_schema 对应表名不同，或退回单线程假设
[done] single_thread_exit — 从库单线程回放跟不上主库并发写入。方向：开启并行复制(基于 WRITESET 的 slave_parallel_type/workers)、检查从库硬件是否弱于主库、避免主库大批量写。
[done] big_txn_exit — 已并行但仍延迟，疑似大事务阻塞回放或从库资源瓶颈。查是否有超大事务(binlog event 大)、从库 IO/CPU 是否打满(转 host 域)。
[escalate] escalate — 打包升级：SHOW SLAVE STATUS 全量、并行 worker 状态、从库资源、主库写入速率。
