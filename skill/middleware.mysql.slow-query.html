决策树

入口：assess

[check] assess — 确认是否存在慢查询与长事务
· 当 max_query_time > 10 → find_slow
· 当 max_query_time <= 10 → not_slow
· 否则 → escalate
⚠ 先看当前是否有正在跑的长查询(PROCESSLIST 的 TIME)，再翻慢日志——现行长查询才是当下的痛点
⚠ 别只看 Slow_queries 计数(累计值)；关注增长速率与当前 running 状态
[done] not_slow — 当前无明显长查询（最长 {{output.max_query_time}}s）。若历史有慢日志，离线分析 slow log(pt-query-digest)找 TopN。
[check] find_slow — 判断慢的成因（锁等待 / 全表扫描 / 资源）
· 当 waiting_lock > 0 → lock_exit
· 当 sending_data > 0 → scan_exit
· 否则 → resource_exit
⚠ state=Waiting for lock → 锁竞争(转 deadlock/长事务)；state=Sending data → 大量数据检索(索引/全表扫描)；都不是→查资源(CPU/IO)
[done] lock_exit — 慢在锁等待。查阻塞源事务(SELECT * FROM performance_schema.data_lock_waits)与长事务(information_schema.INNODB_TRX)。方向：优化事务粒度、避免大事务、必要时人工
[done] scan_exit — 慢在数据检索(可能缺索引/全表扫描)。用 EXPLAIN 看执行计划(type=ALL 即全表扫描)、检查 WHERE 列是否有索引、是否隐式类型转换导致索引失效。交研发优化 SQL/加索引。
[done] resource_exit — 非锁非扫描，疑似资源瓶颈或整体负载。核对主机 CPU/IO(转 host 域)、缓冲池命中率(Innodb_buffer_pool_read_requests)、是否并发过高。
[escalate] escalate — 打包升级：PROCESSLIST、慢日志摘要、当前长查询的 EXPLAIN、锁等待、资源指标。
