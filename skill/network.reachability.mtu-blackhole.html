决策树

入口：test_small

[check] test_small — 小包是否通（基线）
· 当 loss_pct == 0 → test_large
· 当 loss_pct > 0 → not_mtu_exit
· 否则 → escalate
⚠ MTU 黑洞的典型症状：能 ping 通、能建 TCP 连接、但一传大数据就卡死——因为小包过得去大包被丢且 ICMP 分片需求被拦
⚠ 先确认小包通(排除纯不通)，再测大包，对比才有意义
[done] not_mtu_exit — 小包就丢，不是 MTU 问题(MTU 黑洞的前提是小包通)。转 network/reachability/packet-loss 常规丢包排查。
[check] test_large — 大包+禁分片是否通
· 当 loss_pct == 100 → mtu_confirmed_exit
· 当 loss_pct == 0 → no_mtu_issue_exit
· 否则 → escalate
⚠ 必须带 df-bit(禁止分片)：否则中间设备会分片让大包也能过，就测不出黑洞。df-bit 大包不通=确诊 MTU 黑洞
[done] mtu_confirmed_exit — 确诊 MTU 黑洞：大包(禁分片)不通、小包通。根因是路径上某段 MTU 小于报文大小，且 ICMP '需要分片'报文被防火墙/ACL 拦截(PMTUD 失效)。方向：二分法测出黑洞 MTU 值(逐步减小 size)、
[done] no_mtu_issue_exit — 大包(禁分片)也通，无 MTU 黑洞。若业务仍异常，往应用层或其他方向排查。
[escalate] escalate — 打包升级：大小包 ping 结果、路径 MTU、可疑隧道/VPN 段、ICMP 是否被过滤。
