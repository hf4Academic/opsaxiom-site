决策树

入口：quantify_loss

[check] quantify_loss — 量化丢包率
· 当 loss_pct == 100 → full_loss_exit
· 当 loss_pct > 0 and loss_pct < 100 → trace_hops
· 当 loss_pct == 0 → no_loss_exit
· 否则 → escalate
⚠ 设备自身 ping 用软件转发、优先级低，丢几个包未必代表转发面丢包——大样本(100 包)才有统计意义
⚠ 间歇丢包最难：先确认是稳定丢还是偶发。偶发丢包常与拥塞/微突发相关，不一定是故障
[done] no_loss_exit — 本次探测无丢包，可能问题已恢复或为瞬时。若业务仍报丢包，加大探测样本或换源/目标复现。
[done] full_loss_exit — 100% 丢包=完全不通。转 network/reachability(路由/接口)或检查目标是否存活。这不是'丢包'而是'不通'，排查方向不同。
[check] trace_hops — 逐跳追踪定位丢包段
· 当 any(hops[].loss_pct > 50) → hop_loss_exit
· 否则 → interface_check
⚠ 中间跳的丢包/高延迟可能是该跳对 ICMP 限速(control-plane policing)，不代表转发丢包——要结合业务流量验证
[done] hop_loss_exit — 定位到某跳开始丢包。在该跳设备上查其入/出接口错包(转 interface-errors)与拥塞。注意排除 ICMP 限速造成的假象：用实际业务端口验证。
[done] interface_check — 逐跳未见明显单跳丢包，可能是路径上均匀丢包(拥塞)或非对称路由。检查沿途接口计数器与队列丢弃，核对往返路径是否一致。
[escalate] escalate — 打包升级：丢包率、traceroute 逐跳结果、源目、沿途接口计数器。
