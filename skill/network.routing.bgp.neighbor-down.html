决策树

入口：peer_state

[check] peer_state — 确认邻居当前状态
· 当 state == 'Idle' and admin_state == 'shutdown' → admin_shutdown_found
· 当 state == 'Idle' → check_reachability
· 当 state == 'Active' or state == 'Connect' → check_reachability
· 当 state == 'OpenSent' or state == 'OpenConfirm' → check_session_params
· 当 state == 'Established' → flapping_check
· 否则 → escalate
⚠ Idle 和 Active 的含义与直觉相反：Active 表示正在主动尝试 TCP 连接，不是'正常'
⚠ 先看状态机卡在哪一层再动手：TCP 层问题查连通性，OPEN 层问题查参数
[done] admin_shutdown_found — 邻居处于管理性 shutdown（人为关闭）。这是配置意图而非故障——请先与变更记录/同事核实关闭原因，不要擅自 no shutdown。
[check] check_reachability — 检查到邻居的 TCP/IP 可达性
· 当 loss_pct == 100 → check_interface
· 当 loss_pct == 0 → check_tcp_179
· 当 loss_pct > 0 and loss_pct < 100 → escalate
· 否则 → escalate
⚠ 必须指定 source 为 BGP update-source 地址——默认出接口 ping 通不代表 BGP 会话路径通
[check] check_interface — 检查物理接口与直连
· 当 link_state != 'up' → physical_layer_exit
· 当 link_state == 'up' and delta(input_errors, 60s) > 0 → physical_layer_exit
· 当 link_state == 'up' → escalate
· 否则 → escalate
[done] physical_layer_exit — 根因在物理层/链路层（接口 down 或错包增长）。请转入 network/physical 域的对应 Skill（光模块/线缆排查），BGP 会话会在链路恢复后自愈。
[check] check_tcp_179 — ICMP 通但会话不建立——检查 TCP 179 与控制面策略
· 当 count(lines) > 0 → acl_suspect
· 当 count(lines) == 0 → check_session_params
· 否则 → check_session_params
⚠ 还要考虑对端防火墙与 GTSM/TTL 安全（ebgp-multihop / ttl-security 配置不对称）
[done] acl_suspect — 存在涉及 TCP/179 的过滤策略（详单见上）。修改 ACL 属于安全策略变更，超出本 Skill 授权范围——请走 network/security-policy 域流程，并与安全团队确认该条目意图。
[check] check_session_params — 会话参数核对（AS 号 / 认证 / 能力协商）
· 当 any(lines[] matches 'bad AS|OPEN.*error|AS number') → config_mismatch_exit
· 当 any(lines[] matches 'MD5|authentication|TCP.*fail') → config_mismatch_exit
· 当 any(lines[] matches 'notification|hold time|Cease') → ask_soft_reset
· 否则 → ask_soft_reset
[done] config_mismatch_exit — 日志显示会话参数不匹配（AS 号/认证/能力，证据见上）。这需要和对端管理员核对预期配置后走变更流程修正——单侧盲改无意义且危险。已生成核对清单。
[ask] ask_soft_reset — 是否尝试软复位会话
[action] soft_reset — BGP 会话软复位
[check] flapping_check — 当前 Established——判断是否抖动（flapping）
· 当 flap_count_24h >= 3 → escalate
· 当 flap_count_24h < 3 → done_recovered
· 否则 → done_recovered
[done] done_recovered — 会话当前 Established 且无频繁抖动记录，可能在你介入前已自愈。建议保留本次采集的日志证据，观察 24h。
[done] done_ok — 软复位后会话恢复 Established，前缀数回到基线。建议：记录本次根因证据链；若同一邻居 7 天内再次中断，直接走升级路径。
[escalate] escalate — 打包升级：设备型号/版本、状态机卡点、可达性结论、日志证据、已走路径。
