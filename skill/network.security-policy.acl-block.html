决策树

入口：find_applied_acl

[check] find_applied_acl — 确认接口上应用了哪些 ACL
· 当 count(applied_acls) >= 1 → check_hits
· 当 count(applied_acls) == 0 → no_acl_exit
· 否则 → escalate
⚠ 先确认接口方向(in/out)上确实挂了 ACL——被拦不一定是 ACL，也可能是路由/NAT。别一上来就怀疑 ACL
⚠ 本 Skill 只诊断到'哪条 ACL 命中/拦截'，不修改 ACL——改 ACL 是安全策略变更，必须走 network/security-policy 变更流程并经安全团队
[done] no_acl_exit — 该接口方向未应用 ACL，不通的原因不在此。转 network/reachability 或 routing 排查，或检查其他方向/其他设备的 ACL。
[check] check_hits — 查 ACL 命中计数定位拦截条目
· 当 deny_hit_count > 0 → deny_hit_exit
· 当 not any(entries[].hits > 0) → implicit_deny_exit
· 否则 → escalate
⚠ 看 deny 条目的命中计数增长即可定位是哪条在拦。注意每条 ACL 末尾隐含 deny any——命中不到任何 permit 也会被默认拒
[done] deny_hit_exit — 定位到明确的 deny 条目在拦截该流量(命中计数增长)。这条 deny 是有意还是误配需人工+安全团队判断—— 本 Skill 到此为止，不擅自改。已生成：命中的 ACL 条目、方向、流量特征，供变更评审。
[done] implicit_deny_exit — 无显式 deny 命中，但流量也没命中任何 permit——被 ACL 末尾的隐含 deny any 拒绝。需要人工评估是否补一条 permit(走安全变更流程)。同样不擅自改。
[escalate] escalate — 打包升级：接口应用的 ACL、各条目命中计数、被拦流量五元组、方向。
