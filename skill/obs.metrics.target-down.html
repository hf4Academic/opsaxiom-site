决策树

入口：read_target

[check] read_target — 读取 target 状态与最近抓取错误
· 当 up > 0 → up_exit
· 当 last_error_conn_refused > 0 → refused_exit
· 当 last_error_timeout > 0 → timeout_exit
· 当 up == 0 → other_error_exit
· 否则 → escalate
⚠ target down 先看 Prometheus 自己记录的 lastError——connection refused / timeout / 401 / no such host，错误类型直接指向根因，别盲猜
⚠ 别只在被抓端排查：也可能是 Prometheus 侧网络策略/防火墙不让它到 exporter，两端都要看
[done] up_exit — 该 target 当前 up，可能是瞬时抓取失败已恢复。观察是否周期性 down(GC 停顿/exporter 偶发卡)，必要时调 scrape_timeout。
[done] refused_exit — 抓取被拒(connection refused)，多为 exporter 进程没在跑或端口/监听地址不对。方向：登目标机确认 exporter 进程与监听端口(ss -ltnp)、防火墙/安全组是否放行 Prometheus 源、容器是否暴
[done] timeout_exit — 抓取超时(timeout)，exporter 在但响应慢或网络拥塞。方向：手动 curl exporter 端点看耗时、检查目标机负载(转 host/compute/load-high)、exporter 采集项过多导致单次抓取慢则拆分或调
[done] other_error_exit — 抓取失败但非拒绝/超时(如 401/403 鉴权、TLS 握手失败、no such host)。方向：核对 scrape_config 的鉴权/TLS/目标地址解析，按 lastError 原文定位。
[escalate] escalate — 打包升级：target 的 up 值、lastError 原文、scrape_config、目标机 exporter 进程与端口状态。
