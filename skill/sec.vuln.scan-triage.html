决策树

入口：triage_vulns

[check] triage_vulns — 按可利用性与可修复性分诊漏洞
· 当 kev_count > 0 → kev_exit
· 当 critical_count > 0 and has_fix_critical > 0 → fixable_crit_exit
· 当 critical_count > 0 → crit_nofix_exit
· 当 high_count > 0 → high_exit
· 当 high_count == 0 → clean_exit
· 否则 → escalate
⚠ 别按 CVSS 分数排队修：优先级看'是否在被实际利用'(CISA KEV/有公开 EXP) 和'是否有补丁'——一个 KEV 中危比一个无 EXP 的 critical 更该先修
⚠ 有补丁的 critical 先修(改动小见效快)；无补丁的 critical 走缓解(隔离/WAF/关闭受影响功能)而非干等
⚠ 扫描器常有误报与不可达路径漏洞：结合是否真实引入、是否在攻击面上，别被数字吓到全量停机打补丁
[escalate] kev_exit — 存在已知被利用漏洞(KEV，{{output.kev_count}} 个)——最高优先级，攻击者正在野利用。立即：定位受影响资产与攻击面暴露情况、按应急打补丁或缓解、检查是否已被利用的迹象。已生成紧急处置清单。
[done] fixable_crit_exit — 有可修复的 critical 漏洞({{output.critical_count}} 个 critical，含可用补丁)。方向：优先对暴露在攻击面的资产打补丁(改动小见效快)，按变更流程灰度验证后推广。修复后重扫确认(转 sec/vuln
[done] crit_nofix_exit — 有 critical 漏洞但暂无补丁。方向：走缓解而非干等——网络隔离/收敛攻击面、WAF/虚拟补丁、关闭受影响功能、加强监控；跟踪上游补丁进度。
[done] high_exit — 无 critical/KEV，但有 high 漏洞({{output.high_count}} 个)。方向：评估是否在真实攻击路径上，纳入常规修复排期按补丁可得性推进，不必应急。
[done] clean_exit — 未见 critical/high 或 KEV 漏洞，风险可控。建议：确认扫描覆盖完整(镜像/主机/依赖)、纳入常态化扫描与基线对比。
[escalate] escalate — 打包升级：critical/high 数量、是否含 KEV、critical 是否有补丁、受影响资产与攻击面暴露，供安全团队定优先级。
